# 代码风格

程序有双重目的：一是给编译器下指令生成可执行文件；二是记录系统的行为意图，供人类（用户、维护者、律师等）阅读。因此，所有代码的第一条铁律就是：**代码必须是可读的**

> [!TIP]
> 代码可读性的困难在于"可读"没有统一标准。教材从人类的生理限制出发给出了一个参考：**短期记忆和视野范围都很小**。Linux 内核编码风格（Torvalds et al. 1996）的核心假设是：一个编程窗口大约 $30$ 行 $\times$ $80$ 列 $=$ $2400$ 个字符，超出这个范围的东西都得靠人脑记忆

> [!NOTE]
> 编码风格不是品味问题，而是**文化问题**。不要争论"哪种风格更好"，要尊重已有项目的习惯。每个已有项目都构成自己的**文化空间**：你创建自己的项目时可以定规则，但不能偏离社区共识太远

## 格式化

C 编译器对格式几乎无感。把整个程序写成一行、所有标识符都用 `l` 和 `1` 组成，编译器照样能编译。格式化完全是为人类服务的

> [!TIP]
> 选择一套**一致**的空白和文本格式化策略。一致性比"哪种风格更好"重要得多

格式化涉及的内容包括：

+ 缩进
+ 括号（`{}`、`[]`、`()`）的放置位置
+ 运算符前后的空格
+ 行尾空格
+ 多余的空行

人眼和大脑有固定习惯，必须保持一致才能高效工作。教材中使用的格式化规则示例如下

**1. 前缀式大括号：`{` 放在行尾（K&R 风格）**

```c
void foo(int x) {
    // ...
}
```

**2. 类型修饰符/限定词左绑定**

```c
char const *p;  // const 靠左靠类型
```

**3. 函数 `()` 左绑定，但条件关键字后加空格**

```c
foo(x);        // 函数调用紧贴
if (x)         // if 和 ( 之间有空格
for (...)      // for 和 ( 之间有空格
```

**4. 三元表达式 `?` 和 `:` 两边加空格**

```c
int abs = x > 0 ? x : -x;
```

**5. 标点符号（`:` `,` `;`）前面不加空格，后面加空格或换行**

这些规则本身没有绝对价值，它们是**视觉辅助工具**，帮助你和协作者一眼读懂新代码。它们不是要你手动敲出来的，而是应该由工具自动完成

> [!WARNING]
> 不要手动调格式，用工具。让编辑器自动格式化你的代码。大项目中应该用命令行工具（如 `clang-format`、`astyle`）强制格式化策略，否则版本对比（`diff`）会很痛苦

## 命名

命名有两个维度：**技术限制** 和 **语义约定**，人们经常把它们混为一谈

> [!TIP]
> 对所有标识符选择**一致**的命名策略。标识符包括：类型（`struct` 或非 `struct`）、结构体和联合体成员、变量、枚举、宏、函数、函数式宏

### 技术限制

C 中标识符的命名空间很复杂。头文件和宏定义的交互会产生意外问题。教材给了一个看似无害的例子

```c
double memory_sum(size_t N, size_t I, double strip[N][I]);
```

这个函数有四个隐患

| 标识符        | 隐患                                                                 |
| :------------ | :------------------------------------------------------------------- |
| `N`           | 大写标识符，别人可能定义宏 `N` 为某个大数字                          |
| `I`           | 被 `<complex.h>` 占用（表示虚数单位 $\sqrt{-1}$）                   |
| `strip`       | 以 `str` 开头，C 标准库保留了所有 `str` 开头的标识符                 |
| `memory_sum`  | 未来可能被 C 标准抢走                                                |

> [!TIP]
> 在头文件中可见的所有标识符必须是**合规的**。"合规"意味着遵守 C 标准对保留标识符的限制

C 标准中，标识符是 **reserved** 的情况（即不能擅自重新定义）

+ 以 `_` 开头 + 第二个字符是 `_` 或大写字母 $\rightarrow$ **语言扩展保留**，如 `__FILE__`、`_Bool`
+ 以 `_` 开头 $\rightarrow$ **文件作用域标识符**和 `enum`/`struct`/`union` 标签保留
+ 宏名 $\rightarrow$ **全大写**（约定俗成）
+ 所有预定义有意义的标识符都保留：标准库函数、`str` 开头的（如 `strcpy`）、`E` 开头的（如 `EINTR`）、`_t` 结尾的（如 `size_t`）等

> [!WARNING]
> 违反这些规则可能多年不出问题，但某天换个编译器、系统升级、或者 C 标准引入新标识符时，代码就突然编译失败了

> [!TIP]
> 不要污染全局标识符空间。只暴露 API 接口需要的类型和函数。一个好策略是给库加**命名前缀**：例如 POSIX 线程 API 用 `pthread_` 前缀，教材作者的工具库 P99 用 `p99_`/`P99_`（公开）和 `p00_`/`P00_`（内部）

两个容易被忽略的冲突来源

+ **结构体成员名** $\rightarrow$ 标准结构体成员通常带前缀，如 `struct timespec` 的 `tv_sec`（防止用户定义宏 `sec` 冲突）
+ **函数参数名** $\rightarrow$ 头文件中的参数名要小心，使用带前缀的名字更安全

例如，将前面的函数改写为安全版本

```c
double p99_memory_sum(size_t p00_n, size_t p00_i,
                      double p00_strip[p00_n][p00_i]);
```

### 语义约定

名字必须**可辨认**且能**快速区分**。教材给了一张对比表

| 名字               | 可辨认 | 可区分 | 快速  | 说明                               |
| :----------------- | :----: | :----: | :---: | :--------------------------------- |
| `lllll1llOll`      |   ✗    |   ✗    |   ✗   | 容易与数字 1、字母 l 混淆          |
| `myLineNumber`     |   ✓    |   ✓    |   ✗   | 太长，占据宝贵的行宽               |
| `n` / `m`          |   ✓    |   ✓    |   ✓   | 单字母，作用域小，声明就在眼前     |
| `ffs` / `clz`      |   ✗    |   ✓    |   ✓   | 缩写不直观，含义不明确             |
| `lowBit` / `highBit`|   ✓   |   ✓    |   ✓   | 语义清晰，快速可辨                 |
| `p00_orb` / `p00_urb`|  ✓    |   ✓    |   ✓   | 前缀隔离 + 有意义的词根            |

常见的多词组合方式

| 方式              | 示例                  | 说明                                     |
| :---------------- | :-------------------- | :--------------------------------------- |
| CamelCase         | `myLineNumber`        | 内部大写，稍微遮挡视线                   |
| snake_case        | `my_line_number`      | 下划线分隔，C 社区最常用                 |
| Hungarian notation| `szName`              | 类型编码前缀，可读性差，API 变了就乱了   |

> [!NOTE]
> 没有哪种命名规则有绝对价值。务实为上，选择适合项目和团队的风格

根据标识符的类型，命名策略也不同

| 标识符类型     | 命名策略               | 示例                                        |
| :------------- | :--------------------- | :------------------------------------------ |
| 类型名         | 表达一个**概念**       | `struct timespec` $\rightarrow$ 时间；`size_t` $\rightarrow$ 尺寸 |
| 全局常量       | 表达一个**特殊工件**   | `M_PI` $\rightarrow$ $\pi$；`SIZE_MAX` $\rightarrow$ 平台最大尺寸 |
| 全局变量       | 表达一种**状态**       | `toto_initialized` $\rightarrow$ 已初始化   |
| 函数/函数式宏  | 表达一个**动作**（动词）| `strcmp` $\rightarrow$ 比较字符串；`matrixMult` $\rightarrow$ 矩阵相乘 |

> [!TIP]
> 文件作用域标识符必须是**自解释的**。类型名说概念，函数名说动作，变量名说状态。如果想不出一个概念来命名某个数据结构、枚举或算术类型，应该重新审视设计

## 国际化

这一节讨论：代码中的标识符和注释能不能用非英语语言？

> [!NOTE]
> 英语确实是计算机行业的 lingua franca，但不要把它当作理所当然。在非拉丁字母的文化圈中，用自己语言的脚本写变量名和注释完全合理。例如
>
> ```c
> long année = 1990L; // Année de l'écriture de l'œuvre
> ```
>
> 如果你看不懂，问题不在写代码的人，而在你的工具和环境

项目的自然语言应该为大多数参与者服务。在西方社会，通常意味着使用英语，但这不是唯一的正确答案

### C23 之前

技术上允许在标识符中使用 Unicode 字符，但必须用 `\u` 转义写法

```c
long ann\u00E9e = 1990L; // Ann\u00E9e de l\u2019\u00E9criture de l\u2019\u0153uvre
```

这完全违背了用本语言写代码的初衷。工具链需要先转换编码再喂给编译器，几乎没人用

### C23 之后

C23 直接引用 Unicode 标准 UAX #31（Unicode Identifier and Pattern Syntax），规则变得清晰。标识符可以直接使用 Unicode 字符

但 Unicode 有一个问题：同一个字符可能有多种表示。例如字母"é"

+ 直接编码：`\u00E9`
+ 分解写法：`\u0065`（e）+ `\u0301`（重音符号  ´）

C23 采用 **NFC（Normalization Form C，规范化形式 C）** 来解决这个问题：先分解成所有基础字符和附加符号，再重新组合。大多数情况下结果就是你输入的那个字符

但有些字形相同但编码不同的字符会被合并

| 字符                    | 原始编码    | NFC 后      | 说明                         |
| :---------------------- | :---------- | :---------- | :--------------------------- |
| Å（拉丁 A 上加圈）      | `\u00C5`    | `\u00C5`    | 不变                         |
| Å（埃格斯特朗符号）      | `\u2126`    | `\u00C5`    | 被合并到拉丁 A 上加圈        |
| Ω（希腊大写 Omega）      | `\u03A9`    | `\u03A9`    | 不变                         |
| Ω（欧姆符号）            | `\u2126`    | `\u03A9`    | 被合并到希腊 Omega            |

> [!TIP]
> 字母字符只有在 NFC 规范化后**映射到自身**时，才允许用于标识符。换句话说，只使用直接来源于自然语言的字母，或与所有自然语言明显不同的字母（如 $\Re$、$\Im$ 表示实部和虚部）

> [!WARNING]
> 不同语言中可能有字形相同的字母，比如希腊大写 Alpha "Α" 和拉丁大写 "A"，肉眼无法区分。同样，Unicode 的 Decimal Digit 类别中也有不同字形的数字。**只有在字形明显不同时，才在标识符中使用不同字母表的字母或数字变体**

> [!NOTE]
> C23 仍然不支持用上下标数字作为标识符（如 $a_1$、$a_2$）。编译器可能作为扩展允许，但**不可移植**
